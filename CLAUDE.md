# CLAUDE.md — Korvun

Operating rules for working on this repository. These are non-negotiable.
Read this file in full at the start of every session and follow it strictly.

## Project

Korvun — Kernel for Orchestrated Routing, Versatile Unified Nodes.
A single Go binary acting as messaging gateway + multi-model router +
multi-brain orchestrator, with a configurable dispatch policy engine
(privacy/cost/consensus) driven by a no-code visual builder. Self-hosted,
cross-platform (Linux/Windows/macOS, x86-64/ARM64), same binary from a
Raspberry Pi to the cloud.

Go module path: `github.com/Sebastian197/korvun`

## Language

- Conversation with the user: **Spanish**.
- Code, identifiers, comments, commit messages, docs in repo: **English**
  (industry standard).

## Commit messages — CRITICAL

- Commit messages MUST NOT contain any reference to Anthropic or Claude.
  No co-author trailers, no "Generated with" / "Generated by" lines,
  no AI attribution of any kind.
- Use Conventional Commits: `feat:`, `fix:`, `test:`, `docs:`, `refactor:`,
  `chore:`, etc.
- SemVer for versioning.

## Non-negotiable workflow (per phase, not just per stage)

Work proceeds in strict granularity: **Stage → Phase → Task**. Do not advance
to a new phase without completing the full cycle below. Do not advance to a new
stage until all its phases are closed. If a task is too large to grasp in one
go, split it.

For every phase:

1. **Context7 first.** Before writing ANY code that uses a library, framework,
   SDK or external API, consult its version-specific documentation via Context7.
   Never program against an external API from memory. If the verified docs are
   not available, say so instead of inventing signatures.
2. **Tests first (TDD).** Write the test suite that defines the contract BEFORE
   the implementation, and confirm it fails (red). Only then write the minimum
   code to make it pass (green).
3. **Implementation.** Only the code needed to make the tests pass.
4. **Quality gate.** `make quality` must be green (lint + vet + tests +
   coverage) over the WHOLE suite, not just the new code, before closing.
5. **Documentation.** Update stage docs, ADRs, and the master document.

## Go standards

- `gofmt` + `goimports` mandatory.
- `golangci-lint` must pass (govet, staticcheck, errcheck, gosec).
- Wrap errors with `%w`.
- `context.Context` on every cancellable operation.
- No mutable global state.
- No `panic` in normal paths.
- Tests are table-driven and run with `-race`.
- Coverage ≥ 85% in core packages; ≥ 90% in `policy`, `router`, `envelope`,
  `brain`.
- Every exported symbol has a godoc comment.
- Each source file starts with:
  `// Copyright 2026 Sebastián Moreno Saavedra`
  `// SPDX-License-Identifier: Apache-2.0`

## TypeScript / frontend standards (when the UI stage is reached)

- TypeScript `strict`, no implicit `any`, explicit prop types.
- ESLint + Prettier.
- Vitest + React Testing Library (unit/component); Playwright (e2e critical).
- Stack: React + TypeScript + Vite + Tailwind + React Flow; desktop via Wails.

## Dependencies

- Prefer the Go standard library whenever reasonable.
- Every external dependency requires a justifying ADR in `docs/adr/` AND
  Context7 verification before adoption.

## Documentation layout

- ADRs: `docs/adr/`

## Security & operation

- Secrets only via environment / secret manager; never in the repo.
- Input validation at every boundary; TLS on exposed endpoints.
- 12-factor principles.
- Observability: structured logging (`slog`), Prometheus metrics,
  OpenTelemetry traces.

## How to behave

- Move one task at a time. Pause for confirmation before advancing phases.
- Do not skip the red-test step "to save time".
- If asked to use a library and Context7 docs are not verified, stop and say so.
